## Execution Protocol

Execute every task through this exact sequence:

### Phase 1: Comprehension

1. **Parse the task** — Identify exactly what is being requested
2. **Identify task type** — Determine if this is system, design, or combined
3. **Load skills** — Execute the mandatory skill loading based on task type
4. **Clarify ambiguity** — If any requirement is unclear, ask immediately. Do not assume.

### Phase 2: Execution

1. **Plan minimally** — Know what files you will touch before touching them
2. **Execute precisely** — Write clean, production-ready code on the first pass
3. **Stay scoped** — Complete only what was requested. No feature creep. No "improvements" beyond scope.
4. **Handle errors** — If something fails, diagnose and fix. If stuck after 2 attempts, surface the blocker.

### Phase 3: Verification

1. **Lint the code** — Run linting. Fix all issues. No exceptions.
2. **Type check** — Ensure TypeScript compiles without errors
3. **Test if applicable** — Run relevant tests if they exist
4. **Visual verification** — For UI work, confirm the result matches expectations

### Phase 4: Completion

1. **Summarize deliverables** — List exactly what was created or modified
2. **Note decisions** — Document any judgment calls made during execution
3. **Flag concerns** — Highlight anything that needs attention or future work

## Code Quality Standards

Every line of code you write MUST meet these standards:

1. **Type Safety** — Full TypeScript coverage. No `any` types. Explicit return types.

2. **Clean Architecture** — Proper separation of concerns. Single responsibility.

3. **Readability First** — Clear variable names. No nested ternaries. Explicit over clever.

4. **Error Handling** — Graceful failures. User-friendly error messages. No silent failures.

5. **Performance Aware** — Efficient queries. Proper memoization. No unnecessary re-renders.

6. **Accessibility** — Semantic HTML. Proper ARIA labels. Keyboard navigation.

## Anti-Patterns (Never Do These)

- **Skipping skill loading** — Skills contain critical best practices. Always load them.
- **Scope creep** — Do not refactor adjacent code. Do not add unrequested features.
- **Assumptions** — If unclear, ask. Never guess at requirements.
- **Incomplete verification** — Never report complete without linting and type checking.
- **Vague reporting** — "I updated some files" is unacceptable. Be specific.
- **Over-engineering** — Simple problems get simple solutions. No unnecessary abstractions.
- **Silent failures** — If something breaks, report it immediately.
